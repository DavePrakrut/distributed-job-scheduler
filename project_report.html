1. Executive Summary

Codity is a high-performance background task execution engine designed to manage delayed jobs, recurring cron tasks, and prioritizations across a multi-tenant enterprise environment. The core backend uses an atomic database lock model via PostgreSQL SKIP LOCKED queries to guarantee that no task is ever executed twice by concurrent background worker processes.

2. System Architecture

The system consists of four independent components that scale horizontally:

  • API Server: Exposes REST API endpoints for job submission, queue configuration, and authentication.
  • WebSockets Hub: Relays real-time events (job status updates, worker online indicators) to client dashboards.
  • Workers Pool: Standalone Node.js threads that query, lock, and process ready jobs concurrently using custom backoffs.
  • Scheduler: Periodically sweeps scheduled/delayed jobs and converts cron templates into executable task records.

3. Concurrency & Integrity Mechanics

Atomic Job Claiming

To prevent race conditions where multiple workers try to execute the same job simultaneously, the claiming query uses the FOR UPDATE SKIP LOCKED mechanism inside a raw PostgreSQL update block: 

UPDATE "Jobs"
SET "status" = 'RUNNING', "lockedByWorkerId" = $1, "lockedAt" = NOW()
WHERE "id" = (
  SELECT j."id"
  FROM "Jobs" j
  JOIN "Queues" q ON j."queueId" = q."id"
  WHERE j."status" = 'QUEUED'
    AND j."runAt" <= NOW()
    AND q."isPaused" = FALSE
    AND (
      SELECT COUNT(1) FROM "Jobs" rj 
      WHERE rj."queueId" = j."queueId" AND rj."status" = 'RUNNING'
    ) < q."maxConcurrency"
  ORDER BY q."priority" DESC, j."createdAt" ASC
  FOR UPDATE SKIP LOCKED
  LIMIT 1
) RETURNING *;

Optimistic Cron Scheduling Locks

To prevent double-triggering recurring cron schedules in highly available multi-scheduler environments, the scheduler executes an atomic optimistic update before spawning the job: 

const updated = await tx.scheduledJobs.updateMany({
  where: { id: schedule.id, nextRunAt: schedule.nextRunAt },
  data: { lastRunAt: now, nextRunAt }
});
if (updated.count === 0) return; // Skipped (already triggered by another scheduler)

4. Retry Strategies & Dead Letter Queue (DLQ)

Codity implements three retry policies to handle transient task failures:

Strategy Formula Description
Fixed Delay Delay = BaseDelay Retries task after a static configured delay.
Linear Backoff Delay = BaseDelay * Attempt Increments delay linearly with each consecutive attempt.
Exponential Backoff Delay = BaseDelay * (Factor ^ (Attempt - 1)) Multiplies delay exponentially to relieve downstream systems.

If a job exceeds the maximum retry limit, it is automatically routed to the Dead Letter Queue (DLQ). The dashboard allows administrators to inspect the full stack trace and manually trigger retries.
